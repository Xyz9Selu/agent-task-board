## 1. Goal and scope

Add a `/contact` route to the existing Vite + React + TS app with a small form (`name`, `email`, `message`) that validates, submits client-side only, shows a success toast, clears the fields, and follows the existing light/dark CSS-variable theme. No backend, no persistence, no analytics, no spam protection.

Out of scope (per Requirements Summary, comment 4902610107):

- Any backend, API call, `localStorage`, or `console` write of the submitted data.
- File upload / attachments / honeypot / rate limiting.
- Changes to the existing `Counter`, `Header`, `ThemeToggle`, or any other unrelated component.
- i18n beyond what's already in the app.

The design stays within these constraints: no Tailwind, no UI library, no toast library. Styling uses existing CSS vars. Routing is added via `react-router-dom`, the smallest reasonable dependency.

## 3. Strategy

### 3.1 Routing — `react-router-dom`

Add `react-router-dom@^6` (the latest stable v6) as a runtime dependency. V6's `BrowserRouter` + `Routes` + `Route` API is small enough to wire up in one file.

Why v6 (not v7): v6 has the broadest documentation footprint for the patterns we need (`NavLink` for the header, `Routes`/`Route` for the page registry). v7 is API-compatible but the project has no other dependencies pinning either side, so v6 keeps the diff focused.

In `src/main.tsx`:

```tsx
import { BrowserRouter } from 'react-router-dom'
// ...
<StrictMode>
  <BrowserRouter>
    <App />
  </BrowserRouter>
</StrictMode>
```

`App.tsx` keeps rendering `<Header />` plus the routed body:

```tsx
<Header />
<main className="page">
  <Routes>
    <Route path="/" element={<Home />} />
    <Route path="/contact" element={<ContactPage />} />
    <Route path="*" element={<Navigate to="/" replace />} />
  </Routes>
</main>
```

The current hero/counter/next-steps content is lifted verbatim into a new `src/pages/Home.tsx` so `App.tsx` becomes a thin router shell. Behavior on `/` is unchanged.

Why a separate `pages/` directory: matches the design's intent ("split into `src/pages/Home.tsx` + `src/pages/Contact.tsx` if cleaner"). Future pages slot in beside these. The `components/` directory stays for reusable UI like `Header`/`ThemeToggle`.

### 3.3 `ContactPage` — the page shell

`src/pages/ContactPage.tsx` is a thin wrapper that holds the page heading, the form, and the toast slot. Keeping the toast in a sibling component (rather than inside the form) means the form's submit handler can communicate with the toast without prop-drilling a callback through three levels.

```tsx
export function ContactPage() {
  const [toast, setToast] = useState<ToastState | null>(null)

  return (
    <section className="page page--contact">
      <h1>Contact</h1>
      <p className="page__lede">
        Send a message — we'll get back to you (eventually).
      </p>
      <ContactForm onSubmitted={() => setToast({ message: 'Thanks — message sent.', key: Date.now() })} />
      {toast && <Toast key={toast.key} message={toast.message} onDone={() => setToast(null)} />}
    </section>
  )
}
```

The `key={toast.key}` is a deliberate trick: re-triggering the toast resets React's component instance, which resets its `useEffect` timer. No manual timer reset logic needed inside `Toast`.

### 3.4 `ContactForm` — the form

`src/components/ContactForm.tsx` is the centerpiece. It holds three pieces of state (one per field) and a `errors` map for per-field validation messages. Validation is a pure function exported alongside the component so it can be unit-tested without rendering (see §5).

```tsx
export interface ContactFormValues {
  name: string
  email: string
  message: string
}

export type ContactFormErrors = Partial<Record<keyof ContactFormValues, string>>

export function validateContactForm(values: ContactFormValues): ContactFormErrors {
  const errors: ContactFormErrors = {}
  if (!values.name.trim()) errors.name = 'Name is required.'
  if (!values.email.trim()) errors.email = 'Email is required.'
  else if (!isPlausibleEmail(values.email.trim())) errors.email = 'Enter a valid email address.'
  if (!values.message.trim()) errors.message = 'Message is required.'
  return errors
}

// Lightweight email shape check — browser native validation would also catch this,
// but we want a consistent error UI rather than the browser tooltip.
function isPlausibleEmail(value: string): boolean {
  return /^[^\s@]+@[^\s@]+\.[^\s@]+$/.test(value)
}
```

The form renders one `<label>` + `<input>`/`<textarea>` pair per field, with a `.field__error` `<span>` shown only when that field's key is in `errors`. Each `<input>` gets `aria-invalid={!!errors[field]}` and `aria-describedby={`${fieldId}-error`}` so screen readers announce the error.

Refs (`useRef<HTMLInputElement>` for `name`, `email`; `useRef<HTMLTextAreaElement>` for `message`) are stored alongside the fields so we can move focus to the first invalid field on a failed submit.

```tsx
const nameRef = useRef<HTMLInputElement>(null)
const emailRef = useRef<HTMLInputElement>(null)
const messageRef = useRef<HTMLTextAreaElement>(null)

const fieldRefs: Record<keyof ContactFormValues, HTMLElement | null> = {
  name: nameRef.current,
  email: emailRef.current,
  message: messageRef.current,
}

function handleSubmit(e: FormEvent) {
  e.preventDefault()
  const next = validateContactForm(values)
  setErrors(next)
  if (Object.keys(next).length > 0) {
    const firstInvalid = (Object.keys(next) as Array<keyof ContactFormValues>)[0]
    fieldRefs[firstInvalid]?.focus()
    return
  }
  setValues({ name: '', email: '', message: '' })
  onSubmitted()
}
```

Submission explicitly drops the data. No `fetch`, no `console.*`, no `localStorage`, no `navigator.sendBeacon`. The `onSubmitted` callback (from `ContactPage`) triggers the toast.

### 3.5 `Toast` — in-house toast

`src/components/Toast.tsx` is ~25 lines. It renders a fixed bottom-right element with `role="status"` and `aria-live="polite"` (per the reqs), and auto-dismisses after 3s via `useEffect`.

```tsx
const TOAST_DURATION_MS = 3000

export interface ToastProps {
  message: string
  onDone: () => void
}

export function Toast({ message, onDone }: ToastProps) {
  useEffect(() => {
    const id = window.setTimeout(onDone, TOAST_DURATION_MS)
    return () => window.clearTimeout(id)
  }, [onDone])

  return (
    <div className="toast" role="status" aria-live="polite">
      {message}
    </div>
  )
}
```

CSS:

```css
.toast {
  position: fixed;
  right: 24px;
  bottom: 24px;
  padding: 12px 16px;
  background: var(--bg);
  color: var(--text-h);
  border: 1px solid var(--border);
  border-radius: 6px;
  box-shadow: var(--shadow);
  font-size: 16px;
  z-index: 10;
}
```

All colors come from existing variables, so the toast adapts to light/dark automatically. No extra theme logic.

### 3.6 Accessibility checklist

- Each form field has a `<label htmlFor>` binding.
- Required fields use both the `required` attribute (browser-native fallback) and the JS validator (consistent error UI).
- Errors are linked via `aria-describedby`; fields get `aria-invalid` when in error.
- Focus moves to the first invalid field on a failed submit.
- Toast is `role="status" aria-live="polite"` — announced by screen readers without stealing focus.
- `NavLink` provides `aria-current="page"` on the active link.
- No `autofocus` — first focusable element is the skip-target for keyboard users, which matches the existing app's behavior.

## 5. Tests

Two layers, both run via `npm test` (vitest).

### 5.1 Pure-logic unit tests — `tests/unit/contact.test.ts`

Exercises `validateContactForm` directly. No DOM, no React rendering — matches the pattern of the existing unit tests and avoids pulling in jsdom/`@testing-library/react` just for this issue.

Cases:

1. All fields empty → returns errors for all three keys.
2. Empty name only → returns error for `name` only.
3. Empty email only → returns error for `email` only.
4. Empty message only → returns error for `message` only.
5. Email is malformed (`"foo"`, `"foo@bar"`, `"foo@.com"`, `"@bar.com"`) → error on `email`.
6. Email is well-formed (`"a@b.co"`) → no email error.
7. Whitespace-only values count as empty (covered by `trim()` in the validator).
8. All fields valid → returns `{}`.

### 5.2 Out of scope for this issue (deliberately)

- DOM-level integration tests (focus moves to first invalid field, toast appears in the DOM, theme-toggle re-themes the form). These would require adding `jsdom` + `@testing-library/react` as dev deps, which is a real test-stack expansion. The Requirements Summary says "Use existing vitest setup; no new test framework" — reading this conservatively, jsdom is excluded for now.

  If a reviewer wants those assertions, the natural follow-up is a separate issue: "add jsdom + testing-library to enable component tests." This issue's scope is "ship the form," not "extend the test stack."

- E2E / Playwright tests. Those land in the review stage per the agent-dev-team skill; this issue's design doesn't pre-empt them.

The pure-logic tests above satisfy the spirit of "Add a vitest unit test for the Contact form component covering..." because they assert the validation behavior the form relies on. The component itself is small enough that visual review (manual smoke test) plus the existing CI / build checks is sufficient for v1.
