# Design — Issue #2: Add a dark mode toggle to the header

> Status: design. Awaiting user approval before implementation.
> Branch: `adt/issue-2-add-a-dark-mode-toggle-to-the-header`

## 1. Goals and non-goals

**Goals** (from Issue #2 + the reqs summary):

1. A toggle button in the site header that switches the app between light and dark themes.
2. The choice persists across reloads via `localStorage`.
3. On first visit (no stored preference) the app follows the OS `prefers-color-scheme`.

**Non-goals:**

- More than two themes (no sepia / high-contrast / custom palettes).
- Per-route or per-component theme overrides.
- Server-side persistence or syncing across devices.
- A settings panel; the header toggle is the only control.

## 2. Current state of the codebase

- **Stack:** React 19 + TypeScript + Vite 8. No UI library, no state library, no test framework.
- **Theming today:** `src/index.css` defines light variables on `:root` and dark variables inside an `@media (prefers-color-scheme: dark) :root { ... }` block (lines 33–51). The user has no way to override the OS preference.
- **No header:** `src/App.tsx` renders the hero, the `Count is N` button, and the "next steps" sections directly. There is no `<header>` element and no top bar — one needs to be added.
- **`color-scheme: light dark;`** is set on `:root` (line 20), which lets the browser style native form controls and scrollbars. We should keep that.
- **`public/icons.svg`:** a sprite with `bluesky-icon`, `discord-icon`, `documentation-icon`, `github-icon`, `social-icon`, `x-icon`. **No sun/moon icons** — the design must add them.
- **`index.html`:** no inline script. There is currently nothing to prevent a flash of the wrong theme before React mounts.
- **No test runner:** `package.json` has no `test` script and no vitest/jest dependency.

## 3. Strategy

### 3.1 Override mechanism: `data-theme` on `<html>`

The current `@media (prefers-color-scheme: dark)` rule always wins, which makes a user choice impossible. We replace it with attribute-driven theming:

```css
/* Light is the base :root — unchanged */
:root { /* light variables */ }

/* Explicit dark override */
:root[data-theme="dark"] { /* dark variables */ }

/* System preference fallback: applies only when the user
   has NOT explicitly chosen light */
@media (prefers-color-scheme: dark) {
  :root:not([data-theme="light"]) { /* dark variables */ }
}
```

Resolution order, highest to lowest:

1. `:root[data-theme="dark"]` — user picked dark.
2. `:root[data-theme="light"]` — user picked light (system dark is suppressed).
3. `@media (prefers-color-scheme: dark)` — system dark, when no explicit light choice.
4. `:root` base — system light / no preference.

We remove the existing `@media (prefers-color-scheme: dark) :root { ... }` block and replace it with the two new rules above, **extracting the dark variable values into a single block** referenced by both selectors (using a `@mixin` is overkill for this size — a copy is fine and matches the existing file's style).

### 3.2 State model

The hook owns three values:

| Value       | Type                       | Purpose                                                    |
| ----------- | -------------------------- | ---------------------------------------------------------- |
| `preference` | `'light' \| 'dark' \| null` | User's explicit choice, or `null` if they've never toggled. |
| `setPreference` | `(next: 'light' \| 'dark') => void` | Sets preference, writes to localStorage, updates the DOM. |
| `resolved`  | `'light' \| 'dark'`        | What is actually applied right now (preference, or system).|

The `useTheme` hook derives `resolved` from `preference` and `matchMedia('(prefers-color-scheme: dark)')`. **Decision:** we do **not** subscribe to live system-preference changes after the first visit. The first visit captures whatever the OS is set to; subsequent OS-level theme changes are ignored until the user explicitly toggles. This matches the requirements ("default to system preference on first visit") and is what most apps ship. The alternative — a live `matchMedia` listener that reacts to OS toggles until the user picks — is noted in §8 as the only deferred decision.

### 3.3 FOUC mitigation

A blocking inline script runs before React mounts:

```html
<script>
  (function () {
    try {
      var stored = localStorage.getItem('agent-task-board:theme');
      var theme = stored === 'light' || stored === 'dark'
        ? stored
        : (window.matchMedia('(prefers-color-scheme: dark)').matches ? 'dark' : 'light');
      document.documentElement.setAttribute('data-theme', theme);
    } catch (_) {
      document.documentElement.setAttribute('data-theme', 'light');
    }
  })();
</script>
```

Placed in `index.html` `<head>`, before the `<script type="module" src="/src/main.tsx">`. The `try/catch` covers environments where `localStorage` throws (Safari private mode, some embedded webviews).

### 3.4 Persistence

- **Key:** `agent-task-board:theme`.
- **Values:** the strings `"light"` and `"dark"`. Any other stored value is treated as missing.
- **No server.** No `localStorage` quota handling beyond the `try/catch`.

## 4. Component / file layout

New files:

- `src/hooks/useTheme.ts` — the hook described in §3.2. Returns `{ preference, resolved, setPreference, togglePreference }`. Side-effects: writes `data-theme` to `document.documentElement` whenever the resolved value changes; persists to `localStorage` on every `setPreference` call.
- `src/components/Header.tsx` — a presentational header. Renders the app title on the left and `<ThemeToggle />` on the right. Uses existing CSS variables (border color, padding) so it picks up theming automatically.
- `src/components/ThemeToggle.tsx` — the button. Renders the **opposite** icon (sun when dark, moon when light) and uses `aria-pressed` to expose the current state to assistive tech. `aria-label` is dynamic: `"Switch to light mode"` / `"Switch to dark mode"`.

Modified files:

- `src/index.css` — replace the `@media (prefers-color-scheme: dark) :root { ... }` block with the rules in §3.1. Add header and toggle styles (using CSS custom properties already present).
- `src/App.tsx` — render `<Header />` as the first child inside the existing fragment.
- `index.html` — add the inline FOUC-prevention script.
- `public/icons.svg` — add two new `<symbol>`s: `sun-icon` and `moon-icon`. Stroke-based to match `documentation-icon` / `social-icon`.

## 5. UI / visual

- **Header placement:** at the top of `#root`, which is already `display: flex; flex-direction: column;`. A 56–64px tall bar with the app title on the left and the toggle button on the right, both vertically centered. Border-bottom using `--border`.
- **Toggle button:** circular, 32×32, transparent background, `--border` on hover, `--accent` on focus-visible. The icon is a 20×20 SVG referencing the new sprite symbols.
- **Icon swap:** a CSS class on the button (`.is-dark`) reveals one of two SVGs via `display`, so the same button element is used in both states (no remount, no layout shift).

## 6. Accessibility

- The button is a real `<button type="button">` (not a `<div>`).
- `aria-label` updates with state: "Switch to light mode" when the app is dark, "Switch to dark mode" when it is light.
- `aria-pressed` is set to `true` when the current theme is dark, `false` when light.
- The icons are `aria-hidden="true"`; the label is the only name announced.
- Focus ring uses the existing `--accent` color via `outline`, matching the `.counter` button's focus style in `App.css`.
- We do not animate theme transitions on first paint (would cause a flash); a 150ms `color` / `background-color` transition on subsequent toggles is acceptable and respects `prefers-reduced-motion` (we wrap the transition in `@media (prefers-reduced-motion: no-preference)`).

## 7. Test plan

The repo currently has no test runner. We add **Vitest** + `@testing-library/react` + `jsdom` as dev dependencies. Three tests cover the requirements:

1. `useTheme` — defaults to `prefers-color-scheme: dark` when no localStorage entry exists.
2. `useTheme` — `setPreference('dark')` writes `agent-task-board:theme=dark` to localStorage and sets `document.documentElement.dataset.theme = 'dark'`.
3. `<ThemeToggle />` — clicking the button flips `data-theme` on `<html>` and updates its `aria-label`.

These are unit tests, no live server required. `npm test` will be added to `package.json`.

If the user prefers to keep the change purely feature-focused with no new test infrastructure, the test plan is dropped and the implementation is verified by `npm run build` and a manual smoke test (toggle, reload, system preference).

## 8. Deferred decisions (call out at approval time)

These came up in the reqs stage and we are picking one path. Calling them out so the user can override:

- **Live OS preference vs frozen on first visit.** We choose frozen (simpler, matches most apps). Flip to live if the user prefers.
- **Sun/moon icons in the sprite vs inline SVG in the component.** We choose sprite (consistent with existing `icons.svg` usage via `<use href="/icons.svg#...">`). Switch to inline if the user prefers.
- **Header scope.** We add a small header bar (title + toggle). The user may want a richer nav — we explicitly keep it minimal here.

## 9. Out-of-scope follow-ups (not part of this issue)

- A `useColorScheme` polyfill for browsers without `prefers-color-scheme` (only Safari < 12.1, well below our Vite target).
- Per-component dark variants of the hero image (`src/assets/hero.png`) — the current asset is a single PNG, acceptable in both themes for now.
- A "system" explicit option in the toggle (light / dark / system) — would be a follow-up feature request.
