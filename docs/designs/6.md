# Design — Issue #6: Build a habit tracker

> Status: **draft**, awaiting `/adt-approve`
> Branch: `adt/issue-6-build-a-habit-tracker`
> Repo: `Xyz9Selu/agent-task-board`

## 1. Goal

Add a small `habit` subcommand to the `adt` CLI that lets the user keep a daily checklist of habits. Single-user, single-machine, JSON-on-disk — fully standalone from the Issue/stage worker flow.

Three subcommands, defined in the Requirements Summary (comment 4861478085):

- `adt habit add <name>`
- `adt habit done <name>`
- `adt habit list`

Out of scope: streaks, counts, history views, notifications, cadence, sync, web UI, machine-readable output, SQLite, integration with the worker flow.

## 2. Where it lives

New module `src/habits.ts` exposes a small, pure-ish API for reading and mutating the habit store. `src/cli.ts` wires the three subcommands through Commander.

Rationale for a separate module: `cli.ts` already calls into `store.ts`, `config.ts`, `worker.ts`, etc. Keeping `habits.ts` isolated means the habit logic is testable without Commander, and there's a single place to add later features (cadence, streaks) without bloating `cli.ts`.

```
src/
├── habits.ts        # NEW — load/save habits.json, add/done/list operations
├── cli.ts           # MODIFIED — three new subcommands under `adt habit`
src/...             # unchanged
tests/
├── unit/
│   ├── habits.test.ts   # NEW — round-trip storage, idempotency, date logic
tests/...            # unchanged
docs/
├── designs/
│   └── 6.md         # THIS FILE
├── user-manual.md   # MODIFIED — short usage block under ## adt habit
```

No new runtime dependencies. We only use `node:fs`, `node:path`, `node:os` (all already used in `config.ts` and `lock.ts`) and the existing `commander` for CLI wiring.

## 3. Storage

A single JSON file at `<ADT_DIR>/habits.json`, where `ADT_DIR` is `process.env.ADT_DIR || path.join(os.homedir(), '.adt')` — the same resolution rule used by `config.ts` and `lock.ts`. Tests will redirect `ADT_DIR` to a tmp dir (same pattern as `tests/unit/config.test.ts`).

### 3.1 File layout

```json
{
  "habits": [
    {
      "name": "exercise",
      "createdAt": "2026-07-02T13:45:00.000Z",
      "completions": ["2026-07-01", "2026-07-02"]
    },
    {
      "name": "read",
      "createdAt": "2026-07-02T13:45:01.000Z",
      "completions": ["2026-07-02"]
    }
  ]
}
```

- `habits` — array, sorted by `name` (locale-aware, case-insensitive) for deterministic output.
- `name` — unique, non-empty, trimmed. Stored as-given (preserves user's casing).
- `createdAt` — ISO-8601 timestamp from `new Date().toISOString()` at the moment of `add`.
- `completions` — sorted ascending `YYYY-MM-DD` strings (local date). Append-only; no entry is ever removed in v1.

### 3.2 File lifecycle

- File is **absent** before the first `add`. No migration needed.
- On any read/write that touches the file: if missing, treat as `{ habits: [] }`.
- Writes go through a write-temp-then-rename sequence (atomic on POSIX) to avoid torn writes if the process is killed mid-write.
- We do **not** journal or lock — single-user, single-machine. The Requirements Summary explicitly notes "Concurrent writes: best-effort; single-user per machine means race conditions are not a real concern."

### 3.3 Why JSON, not SQLite

- Storage is tiny (a few habits × a few hundred dates = a few KB). SQLite is overkill.
- Hand-readable / hand-editable, which matches the "single user, single machine" intent and lets the user recover from mistakes with a text editor.
- Keeps the habit feature fully decoupled from the worker DB (`state.db`); the Requirements Summary requires no SQLite schema changes.

## 4. Module API — `src/habits.ts`

Pure functions over the file. No globals. Takes the store path as an argument (defaulting to `path.join(ADT_DIR, 'habits.json')`) so tests can inject a tmp path.

```ts
// src/habits.ts (sketch — final names/namespaces in impl)

interface Habit {
  name: string;
  createdAt: string;             // ISO-8601
  completions: string[];         // YYYY-MM-DD, ascending
}

interface HabitsFile {
  habits: Habit[];
}

// File I/O
function loadHabits(filePath?: string): HabitsFile;
function saveHabits(file: HabitsFile, filePath?: string): void;

// Operations
function addHabit(name: string, filePath?: string): Habit;          // idempotent
function markHabitDone(name: string, filePath?: string): Habit;     // idempotent same-day
function listHabitsForToday(filePath?: string): Array<{ habit: Habit; doneToday: boolean }>;

// Helpers (exported for tests)
function todayLocal(): string;                  // YYYY-MM-DD in local TZ
function normalizeName(name: string): string;   // trim + reject empty
```

Key behaviors:

- **`addHabit(name)`**
  - `normalizeName(name)` — `name.trim()`; if empty after trim → throw `EmptyHabitNameError`.
  - If a habit with the same (case-sensitive, trimmed) name exists → return it unchanged. **Idempotent.**
  - Otherwise append `{ name, createdAt: new Date().toISOString(), completions: [] }`, re-sort, save.

- **`markHabitDone(name)`**
  - `normalizeName(name)`. If empty → throw `EmptyHabitNameError`.
  - Find the habit by name. If not found → throw `UnknownHabitError(name)`. (Requirements Summary is explicit: surface the error, don't auto-add.)
  - Compute `today = todayLocal()`. If `completions` already includes `today` → return habit unchanged. **Idempotent.**
  - Else append `today`, sort, save.

- **`listHabitsForToday()`**
  - Load file. Return one row per habit with `doneToday = completions.includes(today)`. Empty store → empty array (CLI prints a friendly empty-state line).

- **`todayLocal()`**
  - `YYYY-MM-DD` from `new Date()`, using local time (`getFullYear/getMonth/getDate`). Why local, not UTC: a user who runs `adt habit done` at 11pm expects it to count for today, even though UTC may have already rolled over.

Errors are typed (subclasses of `Error`) so `cli.ts` can map them to exit codes and friendly messages without string-matching.

## 5. CLI wiring — `src/cli.ts`

Three new subcommands under a shared `habit` Command. Each is one block, mirroring the style of the existing `pause`/`resume` commands.

```ts
const habitCmd = program.command("habit").description("Track daily habits (single-user, local)");

habitCmd
  .command("add <name>")
  .description("Register a habit (idempotent)")
  .action((name: string) => {
    try {
      const h = addHabit(name);
      console.log(`Added '${h.name}'`);
    } catch (e) {
      if (e instanceof EmptyHabitNameError) {
        console.error("error: habit name cannot be empty");
        process.exit(1);
      }
      throw e;
    }
  });

habitCmd
  .command("done <name>")
  .description("Mark a habit done for today (idempotent same-day)")
  .action((name: string) => {
    try {
      markHabitDone(name);
      console.log(`Marked '${name}' done for ${todayLocal()}`);
    } catch (e) {
      if (e instanceof EmptyHabitNameError) {
        console.error("error: habit name cannot be empty");
        process.exit(1);
      }
      if (e instanceof UnknownHabitError) {
        console.error(`error: no such habit: '${name}'. Add it with 'adt habit add ${name}' first.`);
        process.exit(1);
      }
      throw e;
    }
  });

habitCmd
  .command("list")
  .description("Show today's status for all registered habits")
  .action(() => {
    const rows = listHabitsForToday();
    if (rows.length === 0) {
      console.log("No habits yet. Add one with: adt habit add <name>");
      return;
    }
    for (const { habit, doneToday } of rows) {
      const mark = doneToday ? "✅" : "❌";
      console.log(`${mark} ${habit.name}`);
    }
  });
```

Plain-text output only. The ✅/❌ marks render as boxes in environments without emoji font, but `console.log` writes raw UTF-8 either way — and matches the Requirements Summary wording. (`tick`/`cross` boxes in v2 if requested.)

### 5.1 Why a sub-Command, not three top-level commands

Commander's `program.command("habit")` followed by chained `.command(...)`s gives:

- a single `adt habit --help` listing all three subcommands,
- consistent `--help` styling with the rest of the CLI,
- room to grow (e.g. `adt habit remove <name>`) without polluting the top-level namespace.

## 6. Tests — `tests/unit/habits.test.ts`

Mirrors the structure of `tests/unit/config.test.ts` and `tests/unit/store.test.ts`: redirect `ADT_DIR` to a per-test tmp dir, dynamic-import `habits.ts`, exercise the public API.

Test cases (mapping to the Requirements Summary acceptance criteria):

1. `add` creates a habit and persists to disk.
2. `add` is idempotent — re-adding returns the same habit, no duplicate row, `createdAt` unchanged.
3. `add` with an empty / whitespace-only name throws `EmptyHabitNameError`.
4. `add` then `list` shows it with `doneToday === false` on the same call.
5. `done` appends today's date to `completions` (assert `todayLocal()` mock or use a fixed-clock seam).
6. `done` is idempotent same-day — calling twice produces a single entry for today.
7. `list` after `done` shows the habit as `doneToday === true`.
8. `list` does not show yesterday's completion as today's (insert a habit with `completions: ["2020-01-01"]` and verify `doneToday === false`).
9. `done` on an unknown habit throws `UnknownHabitError` mentioning the name.
10. `list` on an empty store returns `[]`.
11. Round-trip: save then reload from disk yields the same shape.
12. Atomic write: a torn write simulation isn't necessary; instead just confirm a tmp file is created during save and renamed over the target. (Covered by inspection of the write path; the test only needs to assert post-conditions.)

For test #5/#7 we need a controllable clock. `todayLocal()` reads from `new Date()`, which is hard to stub in pure ESM without monkey-patching. Options:

- **(a)** Inject `now: () => Date` as an optional parameter on `markHabitDone` / `listHabitsForToday`; defaults to `() => new Date()`. Tests pass a fixed `() => new Date('2026-07-02T13:00:00Z')`. **Picked.**
- (b) Use `vi.useFakeTimers()` from Vitest in the test file. Works but adds a global timer mutation across the suite — easier to accidentally affect other tests.

The injected `now` is optional in the public API; the CLI uses the default.

## 9. Risks & mitigations

- **Clock-skew / midnight edge case.** A `done` call at 23:59:59 vs. 00:00:01 may land on different days. Acceptable; that's exactly what `local` date means, and the user controls when they run it.
- **Disk full / permission denied on `~/.adt/`.** `saveHabits` will throw, the CLI surfaces the underlying error, exits non-zero. The Requirements Summary doesn't call for richer recovery; we follow `console.error` + `exit(1)` like the other commands.
- **Name collisions differing only in case.** Treated as distinct (e.g. `Exercise` vs `exercise`). We don't lowercase — preserving user casing matches the storage shape.
- **Habit name containing shell-unsafe chars.** Out of scope; users quote their arguments. This is consistent with `adt pause owner/repo#42`.
- **Growth of `completions` over years.** A habit marked daily for 10 years is ~3.6 KB. Negligible. No need for compaction in v1.
