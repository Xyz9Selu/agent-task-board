# Design — Issue #17: Habit tracker dashboard page

> Status: **draft**, awaiting `/adt-approve`
> Branch: `adt/issue-17-add-a-habit-tracker-dashboard-page`
> Repo: `Xyz9Selu/agent-task-board`

## 1. Goal

Add a `/habits` dashboard page to the existing Vite + React + TypeScript app so the user can list, add, mark-done, and view a streak count for daily habits in the browser. Data persists in `localStorage`. The page matches the existing app's dark-mode + shared-Header look.

This is the **browser** counterpart to the existing CLI habit store in `src/habits.ts` (issue #6). It is a parallel store, not a sync — out of scope to share data with the CLI's `~/.adt/habits.json`.

## 2. Decisions locked from the reqs stage

Picked up verbatim from the Q&A in comment 4902037298 (defaults confirmed by the user):

Why a parallel `src/habits/` directory (and not extending `src/habits.ts`): the existing `src/habits.ts` is **Node-only** — it imports `node:fs`, `node:os`, `node:path`, and resolves a file path under `~/.adt/`. Importing it from a browser bundle would either fail to resolve at build time or pull `node:fs` into the client output. The two stores share semantics (shape, name rules, date format) but **never share code or data**. This keeps the CLI tests and the browser tests from contaminating each other.

The browser store's data shape is deliberately identical to the CLI's `HabitsFile` so that a future "sync" feature could round-trip JSON across them without translation. The two APIs diverge in one place: the browser store returns a `Habit[]` plus computed `doneToday` + `streak`, while the CLI returns `Array<{ habit, doneToday }>` and never computes a streak.

## 4. Data model

Browser-side, persisted under `localStorage["agent-task-board:habits"]`:

```ts
interface Habit {
  name: string;          // trimmed; case-sensitive unique; never empty
  createdAt: string;     // ISO-8601
  completions: string[]; // YYYY-MM-DD, local-time, ascending, no dups
}

interface HabitsFile {
  habits: Habit[];
}
```

JSON shape on disk (matches `src/habits.ts` §3.1):

```json
{
  "habits": [
    {
      "name": "exercise",
      "createdAt": "2026-07-02T13:45:00.000Z",
      "completions": ["2026-07-01", "2026-07-02"]
    }
  ]
}
```

Notes:

- **Sort order on disk:** ascending by `name` using `localeCompare(..., { sensitivity: "base" })`. Same as the CLI. The browser sorts the same way on read.
- **Completions:** always sorted ascending, never contain duplicates, always local-time `YYYY-MM-DD`. Streak computation can rely on this and skip a sort.
- **No `createdAt` is displayed** in v1. We persist it for free because it costs nothing and any future "history" view will need it.

## 5. Domain module API — `src/habits/store.ts` and friends

All functions in this layer are **pure with respect to storage**: they take a `storage` argument (defaulting to `window.localStorage`) and a `now` argument (defaulting to `() => new Date()`). That keeps the entire layer testable in Vitest under `jsdom` without monkey-patching globals, and matches the pattern used in `src/habits.ts` for the CLI.

### 5.1 `src/habits/today.ts`

```ts
export function todayLocal(now: Date = new Date()): string {
  const y = now.getFullYear();
  const m = String(now.getMonth() + 1).padStart(2, "0");
  const d = String(now.getDate()).padStart(2, "0");
  return `${y}-${m}-${d}`;
}
```

Verbatim from `src/habits.ts`. Used by the store, the streak helper, and the done button. Both modules export the same function under the same name; that way the tests of each can use the same fixture dates.

### 5.2 `src/habits/normalize.ts`

```ts
export function normalizeName(name: string): string {
  return name.trim();
}
export function isEmptyName(name: string): boolean {
  return normalizeName(name) === "";
}
```

Verbatim from `src/habits.ts`. Same trim-and-reject-empty semantics; case is preserved.

### 5.3 `src/habits/store.ts`

```ts
export const HABITS_STORAGE_KEY = "agent-task-board:habits";

export interface Habit { /* ... as above */ }
export interface HabitsFile { habits: Habit[] }

// Read / write
export function loadHabits(storage?: Storage): HabitsFile;
export function saveHabits(file: HabitsFile, storage?: Storage): void;

// Operations — same names as the CLI module, same error names.
export function addHabit(
  name: string,
  storage?: Storage,
  now?: () => Date,
): Habit;                                  // idempotent; throws EmptyHabitNameError
export function markHabitDone(
  name: string,
  storage?: Storage,
  now?: () => Date,
): Habit;                                  // idempotent same-day; throws EmptyHabitNameError / UnknownHabitError
export function listHabits(
  storage?: Storage,
  now?: () => Date,
): Array<{ habit: Habit; doneToday: boolean; streak: number }>;   // includes streak

// Errors (subclasses of Error) so the UI can branch on `instanceof` without
// string-matching.
export class HabitError extends Error { /* ... */ }
export class EmptyHabitNameError extends HabitError { /* ... */ }
export class UnknownHabitError extends HabitError {
  readonly habitName: string;
  /* ... */
}
```

Key behaviors (mirror the CLI module):

- `addHabit(name)`: normalize, throw `EmptyHabitNameError` if empty, return existing habit unchanged if a case-sensitive match exists (idempotent), else push `{ name, createdAt: new Date().toISOString(), completions: [] }`, re-sort by name, persist.
- `markHabitDone(name)`: normalize, throw `EmptyHabitNameError` if empty, throw `UnknownHabitError(name)` if not found, compute `today`, return habit unchanged if `today` already in `completions` (idempotent), else append + sort + persist.
- `listHabits()`: load file, return one row per habit with `doneToday = completions.includes(today)` and `streak = computeStreak(completions, today)`.
- `loadHabits`: missing key → `{ habits: [] }`. Empty string → `{ habits: [] }`. Malformed JSON → `{ habits: [] }` (matches CLI's "tolerates a malformed file as empty" rule; recovery is silent rather than throwing, because the UI cannot show a broken-state view in v1).
- `saveHabits`: write the full file. `try/catch` around `storage.setItem` to handle Safari private mode and quota errors — failures bubble up as a `StorageError` (subclass of `HabitError`) which the UI shows as an inline error.

### 5.4 `src/habits/streak.ts`

```ts
/**
 * Current streak ending today: consecutive daily completions counting back
 * from `today`. Reads 0 if today is not in `completions` (per Q3 decision).
 *
 * Preconditions: `completions` is sorted ascending and deduped.
 */
export function computeStreak(
  completions: string[],
  today: string,
): number {
  if (!completions.includes(today)) return 0;
  const set = new Set(completions);
  let streak = 0;
  let cursor = new Date(today + "T00:00:00");   // local midnight
  while (set.add === undefined || true) {       // always true; see below
    const key = todayLocal(cursor);
    if (!set.has(key)) break;
    streak += 1;
    cursor.setDate(cursor.getDate() - 1);
  }
  return streak;
}
```

The body uses an explicit `while` loop with `todayLocal(cursor)` because iterating calendar days across DST boundaries is safer than adding fixed `86400000` ms — `Date.setDate(getDate() - 1)` is the canonical local-day decrement. We'll rewrite the sketch more clearly in the impl stage; the intent is:

1. If `today` is not in `completions` → return `0`.
2. Otherwise, count `1` for today, then walk back one local day at a time while that day is in `completions`. Stop at the first miss.

Edge cases covered by tests (§7):

- Empty completions → `0`.
- `today` not present but yesterday is → `0` (per Q3(a), no grace day).
- `today` + yesterday + day before → `3`.
- A gap: today, day-before-yesterday only → `1`.
- Repeated out-of-order input is normalized upstream by `markHabitDone`'s sort, so the helper does not defend against unsorted arrays.
- DST spring-forward / fall-back: the loop walks by local calendar day, not by ms, so 23- and 25-hour days are both handled.

### 5.5 `src/hooks/useHabits.ts`

A thin React state hook that owns the in-memory `HabitsFile`, hydrates it from `localStorage` on mount, persists on every mutation, and exposes `add`, `done`, `remove` callbacks plus a derived `rows` array.

```ts
export function useHabits() {
  const [file, setFile] = useState<HabitsFile>(() => loadHabits());

  // Re-derive rows whenever `file` changes. The hook also re-renders on a
  // minute-resolution timer so a long-open tab crosses midnight without a
  // reload — see §10 (Risks).
  const rows = useMemo(
    () => listHabits(file),
    [file],
  );

  const addHabit = useCallback((name: string) => {
    setFile((prev) => {
      const file = loadHabits();
      const habit = addHabit(name);    // helper reads/writes via storage
      return loadHabits();             // re-read so we get the canonical sort
    });
  }, []);

  // ... done, remove similar.

  return { rows, add: addHabit, done: markHabitDone, remove: removeHabit };
}
```

In practice the hook is the only place that calls `loadHabits` / `saveHabits`. The pure module stays pure; React state stays React state. This keeps the helper unit-testable without `renderHook` and avoids a giant test file that drags in `jsdom` for what is really just JSON I/O.

A sketch of the actual implementation:

```ts
export function useHabits() {
  const [file, setFile] = useState<HabitsFile>(() => loadHabits());

  const rows = useMemo(() => listHabits(file), [file]);

  const add = useCallback((name: string) => {
    try {
      const file = loadHabits();
      addHabit(name);                  // mutates + persists
      setFile(loadHabits());
    } catch (e) {
      // re-throw; the AddHabitForm catches and shows an inline message.
      throw e;
    }
  }, []);

  const done = useCallback((name: string) => {
    try {
      markHabitDone(name);
      setFile(loadHabits());
    } catch (e) {
      throw e;
    }
  }, []);

  const remove = useCallback((name: string) => {
    const file = loadHabits();
    file.habits = file.habits.filter((h) => h.name !== name);
    saveHabits(file);
    setFile(file);
  }, []);

  return { rows, add, done, remove };
}
```

### 6.3 HabitsPage — `src/components/habits/HabitsPage.tsx`

```tsx
export function HabitsPage() {
  const { rows, add, done, remove } = useHabits();
  const [error, setError] = useState<string | null>(null);

  const handleAdd = (raw: string) => {
    try {
      add(raw);
      setError(null);
    } catch (e) {
      if (e instanceof EmptyHabitNameError) setError("Habit name cannot be empty.");
      else if (e instanceof HabitError)     setError(e.message);
      else                                 throw e;
    }
  };

  return (
    <section className="habits-page">
      <h1>Habits</h1>
      <AddHabitForm onAdd={handleAdd} />
      {error && <p className="habits-page__error" role="alert">{error}</p>}
      {rows.length === 0 ? (
        <p className="habits-page__empty">
          No habits yet. Add one above.
        </p>
      ) : (
        <HabitList rows={rows} onDone={done} onRemove={remove} />
      )}
    </section>
  );
}
```

The page section reuses `flex-grow: 1` so it sits inside the existing `#root` column layout (header → page → footer spacer) the same way the home counter does today.

### 6.4 AddHabitForm — `src/components/habits/AddHabitForm.tsx`

- A controlled `<input type="text">` + a `<button type="submit">`.
- Submit on Enter or button click. Disabled while the input is empty after trim (still allows the user to submit a whitespace-only string and see the error, which matches the CLI's behavior — but we don't waste a write).
- The parent (`HabitsPage`) handles errors and renders them in an inline `role="alert"` element below the form.

### 6.5 HabitList + HabitRow

Each row, left to right:

```
[ habit name ]   [ 🔥 N ]   [ Done for today ✓ / ✓ done ]   [ × ]
```

- Name: `<span className="habit-row__name">{habit.name}</span>`.
- Streak: small badge to the right of the name, rendered as `🔥 {streak}` when `streak > 0`, otherwise a muted `0`. The fire emoji is decorative — a screen reader would announce "streak 3". (`aria-label="Current streak: 3"`.)
- Done: a single `<button>` whose label flips between **"Done for today"** and **"Done ✓"** depending on `doneToday`. Clicking it calls `onDone(habit.name)`. Disabled after done — to un-mark a completion in v1 the user can edit `localStorage` by hand (matches the "add-only + delete via ×" decision in Q5).
- Delete: a small `<button aria-label="Delete habit">` containing an `×` glyph. Calls `onRemove(habit.name)` after a `window.confirm("Delete 'exercise'?")`. Confirmation prevents accidental clicks; "Cancel" leaves the row intact.

`HabitList` is a semantic `<ul>` with one `<li>` per habit. Each row is flex-aligned with `gap: 12px`.

## 7. Tests

The repo uses **Vitest** (already installed; `npm test` works). The web tests need a DOM, so we add `jsdom` and `@testing-library/react` as dev dependencies. Both are small, well-aligned with the existing toolchain, and standard for React component tests.

### 7.1 Unit tests (no DOM)

These run in plain Node and exercise the pure helpers + the store (with a `Storage` shim).

- **`tests/unit/habits-today.test.ts`** — `todayLocal` returns local-time `YYYY-MM-DD`, zero-pads months and days, mirrors the CLI's tests.
- **`tests/unit/habits-normalize.test.ts`** — `normalizeName` trims; `isEmptyName` returns `true` for whitespace-only and `""`.
- **`tests/unit/habits-streak.test.ts`** — the streak helper across:
  - Empty `completions` → `0`.
  - Only yesterday → `0` (today not present).
  - Today only → `1`.
  - Today + yesterday → `2`.
  - Today + yesterday + 5 days ago, gap day before yesterday → `2` (stops at the gap).
  - Three completions ending in a 30-day gap → `1`.
  - Today + every prior day for 7 days → `7`.
  - DST spring-forward (Mar 8 2026 in US): mark today, yesterday, day-before → `3` (proves we walk by local day, not ms).
  - Out-of-order input: `{ "2026-07-01", "2026-07-03", "2026-07-02" }` with today = `2026-07-03` → `3` (the helper trusts callers; the store always sorts first).
- **`tests/unit/habits-store.test.ts`** — round-trip + error semantics using an in-memory `Storage` shim:
  - Empty store → `addHabit("read")` persists, `listHabits()` returns one row with `doneToday=false`, `streak=0`.
  - Idempotent `addHabit`: re-adding returns the same `createdAt`, file still has one entry.
  - `markHabitDone`: appends today; second call same day is a no-op; calls on an unknown name throw `UnknownHabitError` carrying the name; `EmptyHabitNameError` on `""`.
  - `listHabits` includes `streak` matching the helper.
  - `remove` (via the hook's path, or a helper added for symmetry) deletes the habit and its completions.
  - Corrupt JSON in the storage key → loads as `{ habits: [] }`; subsequent `add` re-creates the file.
  - Case-sensitive dedupe: `addHabit("Exercise")` then `addHabit("exercise")` yields two rows.

### 7.2 Component tests (jsdom)

These live alongside the helpers but use `@testing-library/react` to exercise `<HabitsPage />` end-to-end:

## 10. Risks & mitigations

- **DST & timezones.** Streak walks local calendar days, not `86400000` ms. Tested explicitly on a US spring-forward date.
- **Midnight rollover in a long-open tab.** A user who keeps `/habits` open across midnight will see yesterday's row still labeled "Done for today" until they refresh. We mitigate with a `setInterval` inside `useHabits` that bumps a `today` tick every 60s and re-derives `rows` (cheap; ≤ a few habits). Not free, but the alternative — "show a banner that says refresh to update" — is worse UX.
- **localStorage quota / Safari private mode.** The store wraps `setItem` in `try/catch` and surfaces a `StorageError`. The page shows the error in a `<p role="alert">` above the form. No automatic fallback to `sessionStorage` — v1 stays single-store.
- **Multi-tab editing.** Two tabs open on `/habits`: writes from one will overwrite the other on next reload. We don't add a `storage` event listener in v1 (the behavior is "last write wins", which is the standard for `localStorage`-backed apps and is acceptable for a personal habit tracker).
- **Data shape drift from the CLI.** The two modules share a JSON shape by convention, not by code. We will keep them in sync manually. If a future sync feature lands, we should add a `compat.test.ts` that loads a CLI-generated `habits.json` and asserts the browser store reads it without error.
- **`react-router-dom` v6 vs v7.** We pin to `^6.x` for stability and minimal API surface; v7's data routers are unnecessary for two routes.
- **No SSR / build-time prerender.** The app is pure client-side. The router does not need a static fallback.
