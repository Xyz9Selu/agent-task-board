# Design — Issue #21: `adt doctor` config validation

> Status: **draft**, awaiting `/adt-approve`
> Branch: `adt/issue-21-add-an-adt-doctor-command-for-config-validation`
> Repo: `Xyz9Selu/agent-task-board`

## 1. Goal

Add a read-only `adt doctor` subcommand that validates the local adt configuration and runtime environment. It must not modify the filesystem, post to GitHub, or alter state.db. It is purely diagnostic — a one-shot report that returns exit 0 when everything is healthy and exit 1 otherwise.

Behavior decisions come from the `## adt: grill` and `## adt: reqs` comments on issue #21. The key changes vs. the original issue spec:

- **Token resolution** is env-first (`GITHUB_TOKEN`, then `GH_TOKEN`), config-fallback. The worker adopts the same order so `doctor` validates what `run` actually uses.
- **Required-label set** is `ALL_ADT_LABELS` from `src/labels.ts` (covers grill-running/-waiting, drops the `injection` typo from the issue).
- **Per-repo labels** — pass if **any one** configured repo has the full set; print per-repo results either way.
- **`ccMmPath`** — if the value has no `/`, resolve via `PATH`; if it does, do a direct fs+exec check.
- **No-config behavior** — print the Config check and keep going. `doctor` must work before `setup`.
- **Output** — plain text, no ANSI color, matches `adt status` style.
- **Network failures** — count as a check failure with the underlying reason printed.
- **Parallelism** — `Promise.all` across repos to fit the 5 s budget.

## 3. Token resolution (Q1 follow-on)

`loadConfig()` is intentionally unchanged for backward compatibility. A new exported helper handles the order required by the grill answer.

### 3.1 `src/config.ts` additions

```ts
// Returns the GitHub token from env (preferred) or config (fallback).
// Order matches what the worker will use, so doctor validates reality.
function resolveToken(cfg: Config | null): string | null {
  return process.env.GITHUB_TOKEN
      || process.env.GH_TOKEN
      || (cfg?.githubToken ?? null);
}

function tokenSource(): 'env:GITHUB_TOKEN' | 'env:GH_TOKEN' | 'config' | null {
  if (process.env.GITHUB_TOKEN) return 'env:GITHUB_TOKEN';
  if (process.env.GH_TOKEN)     return 'env:GH_TOKEN';
  return null; // config has no "source" sentinel from outside
}
```

`Config` interface gains no fields — the existing `githubToken` field stays the config fallback.

### 3.2 Worker alignment

`src/worker.ts` has two places that read `config.githubToken`:

- Line 125: `const client2 = createClient(config.githubToken);` (cleanup pass)
- Line 191: `const client = createClient(config.githubToken);` (GitHub scan)
- Line 229: `const client = createClient(config.githubToken);` (waiting-user design)
- Line 242, 251, 273: more of the same
- Line 289: fall-through GitHub scan
- Line 316: `const client = createClient(config.githubToken);` (main issue fetch)
- Line 345: `env: { GH_TOKEN: config.githubToken }` (cc-mm spawn)

All become `resolveToken(config)` (with `null`-guard for safety). This is a behavior change — when `GITHUB_TOKEN` is set in the user's shell, the worker now uses it directly, matching what `doctor` reports. This is the explicit requirement from the grill answer.

## 4. `src/doctor.ts` — check architecture

Each check is a small async function returning a `CheckResult`. The runner collects them, formats the table, and decides the exit code.

```ts
interface CheckResult {
  name: string;
  ok: boolean;
  detail?: string;   // one-line explanation on failure
  subItems?: string[]; // for per-repo results (label coverage)
}

type Check = () => Promise<CheckResult>;

const CHECKS: Record<string, Check> = {
  config: checkConfig,
  token: checkToken,
  repos: checkRepos,
  ccMm: checkCcMm,
  labels: checkLabels,
};
```

### 4.1 `checkConfig()`

- Read `<ADT_DIR>/config.json` (resolved the same way as `loadConfig`).
- If file is missing: `{ ok: false, detail: "no config at <path> — run \`adt setup\`" }`.
- If file exists but `JSON.parse` throws: `{ ok: false, detail: "config is not valid JSON: <parse error>" }`.
- If parsed value is missing `githubToken`, `repos`, or `ccMmPath`: `{ ok: false, detail: "missing required field: <field>" }`.
- Else: `{ ok: true }`.

### 4.2 `checkToken()`

- Call `resolveToken(cfg)` where `cfg` is `loadConfig()` if the file parsed, else `null`.
- If null: `{ ok: false, detail: "no GitHub token found in env or config" }`.
- Else: `await client.rest.users.getAuthenticated()` (or `users.get` for the bot case) via a fresh Octokit. Treat 200 as ✓; treat 401 as `{ ok: false, detail: "token rejected by GitHub (401) — token may be expired or missing scopes" }`. Other errors: `{ ok: false, detail: \`GitHub error: <status> <message>\` }`.
- Use the existing `withGh`-style retry helper from `worker.ts` for 5xx? **No** — `doctor` is one-shot, retries would push past the 5 s budget. One attempt, fail fast.
- Wrap the call in a 4 s `Promise.race` timeout so a hung API doesn't block the rest.

### 4.3 `checkRepos()`

- For each repo in `config.repos[]`, fire `client.rest.repos.get({ owner, repo })` in parallel via `Promise.all` (Q8).
- Per-repo pass/fail. Aggregate: ✓ only when every repo passes.
- 404 → "repo not found or no access (404)"; 403 → "rate limited or forbidden (403)"; other → generic message.
- If `config.repos` is empty (no-config case): `{ ok: false, detail: "no repos configured — run \`adt setup --add owner/repo\`" }`.

### 4.4 `checkCcMm()`

- Read `cfg.ccMmPath` (or default `'cc-mm'` if config missing entirely — see Q5).
- If the value contains no `/`: spawn `which <value>` via `node:child_process.execFile` (no shell), trim trailing newline, then check `fs.existsSync(resolved) && (fs.statSync(resolved).mode & 0o111) !== 0`.
- If the value contains `/`: `fs.existsSync(value) && (fs.statSync(value).mode & 0o111) !== 0`.
- Failure message: `"cc-mm not found — install it or set ccMmPath in config"` (PATH case) or `"cc-mm not executable at <path>"` (path case).
- Wrapped in a 2 s timeout (`Promise.race`) so a missing `which` doesn't hang.

### 4.5 `checkLabels()`

- For each repo in `config.repos[]`, fire `client.rest.issues.listLabelsForRepo({ owner, repo, per_page: 100 })` in parallel.
- Build a `Set<string>` of label names per repo.
- For each repo, compute `missing = ALL_ADT_LABELS.filter(l => !present.has(l))`.
- Per-repo sub-line: `<repo>: <n>/<total> labels present` (with missing list only on failure).
- Aggregate ✓ if **any one** repo is missing zero labels (Q3).
- If `config.repos` is empty: `{ ok: false, detail: "no repos to check labels against" }`.

### 4.6 Runner

```ts
async function runDoctor(): Promise<number> {
  const checks = Object.entries(CHECKS).map(async ([_name, fn]) => fn());
  // Sequentially call so labels/repos share a single config load, but
  // parallelize internal fan-out within each check.
  const results: CheckResult[] = [];
  for (const fn of Object.values(CHECKS)) results.push(await fn());

  // Print table
  for (const r of results) {
    const mark = r.ok ? "✓" : "✗";
    const line = `${mark} ${r.name.padEnd(8)} ${r.ok ? "ok" : r.detail}`;
    console.log(line);
    if (r.subItems) for (const s of r.subItems) console.log(`    ${s}`);
  }
  const allOk = results.every(r => r.ok);
  console.log("");
  console.log(allOk ? "All checks passed." : "Some checks failed. See above.");
  return allOk ? 0 : 1;
}
```

Output style matches `adt status` (line 93–97 of `cli.ts`): no ANSI, padded columns, one check per line.

## 5. CLI wiring (`src/cli.ts`)

A new top-level command, registered between `pause`/`resume` (lines 116–147) and `habitCmd` (line 149) so it groups with other diagnostic-ish commands:

```ts
program
  .command("doctor")
  .description("Validate local config and runtime (read-only)")
  .action(async () => {
    process.exit(await runDoctor());
  });
```

`runDoctor` already returns the right exit code, so the action just delegates. No flags in v1 — the issue scope is a single diagnostic command, and adding flags later doesn't break callers.

## 6. Tests

`tests/unit/doctor.test.ts` covers each check in isolation. Same pattern as `tests/unit/config.test.ts`: redirect `ADT_DIR` to a tmp dir, set/unset env vars in `beforeEach`/`afterEach`, use `nock` for GitHub calls.

Test list:

1. **checkConfig** — missing file fails; valid file passes; malformed JSON fails with parser message; missing `githubToken` fails.
2. **checkToken** — `GITHUB_TOKEN` env wins over config; `GH_TOKEN` is the secondary env; `nock` 401 fails; `nock` 200 passes.
3. **checkRepos** — nock 200 per repo passes; nock 404 for one repo fails the whole check (per-repo sub-line); empty repos fails.
4. **checkCcMm** — bare `cc-mm` resolves via stubbed `which`; explicit path with exec bit passes; explicit path without exec bit fails.
5. **checkLabels** — full label set passes; one missing label fails; two repos, one complete + one partial → overall ✓ (Q3); empty repos → skip.
6. **runner integration** — all checks pass → exit 0, prints "All checks passed."; one check fails → exit 1, prints "Some checks failed."

For `checkCcMm`, the `which` lookup can be mocked by prepending a tmp dir to `process.env.PATH` and dropping a `chmod 0o755 cc-mm` shim in it. No need for `vi.mock` gymnastics.

For `checkToken` and `checkRepos` / `checkLabels`, follow `tests/unit/github.test.ts` — `nock(BASE).get(...).reply(200, [...])` with `nock.cleanAll()` in `beforeEach`.

## 7. Acceptance criteria mapping

| Requirement (from reqs summary)                                 | Where it's met                                     |
|-----------------------------------------------------------------|----------------------------------------------------|
| `adt doctor` runs in <5 s                                       | `Promise.all` per-repo (Q8); 4 s token timeout; 2 s which timeout |
| Each check prints a clear ✓/✗ line plus one-line failure msg   | `runner` formatting, `detail` field                |
| Exit code matches overall pass/fail state                       | `runner` returns 0/1, CLI `process.exit`s it       |
| Human-readable, no JSON dumps                                   | Plain `console.log`; no `JSON.stringify` anywhere  |
| Read-only — no modifications                                    | Only `fs.readFileSync` and network GETs; no writes, no `postComment`, no `state.db` |
| Token source matches worker                                     | Shared `resolveToken()` helper                     |
| Labels match code, not issue spec                               | Imports `ALL_ADT_LABELS` from `src/labels.ts`     |
| Any-one-repo pass rule for labels                               | `checkLabels` aggregate logic                      |
| `ccMmPath` PATH resolution                                      | `checkCcMm` `which` branch                         |
| Works without a config                                          | `loadConfig` is wrapped; missing-file path is reported, other checks still run |

## 8. Out of scope (explicit)

- Auto-fix — doctor never writes anything (no label creation, no config writes).
- New flags (`--json`, `--strict=false`, etc.) — Q7/Q8 settled on defaults; flags can come later.
- Replacing `loadConfig` itself — we add `resolveToken()` alongside, keep `loadConfig` behavior identical.
- Adjusting the worker's GitHub retry helper (`withGh`) — `doctor` is single-shot by design.
